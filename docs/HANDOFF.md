# Source Code Handoff

This document provides everything needed to understand, build, and maintain the BlueBubbles iMessage API codebase.

## Repository Structure

```
bluebubbles-api/
  src/                      Application source (TypeScript)
    config/                 Environment configuration (Zod-validated)
      env.ts                All env vars with defaults and validation
    middleware/              Express middleware
      auth.ts               API key authentication (Bearer token)
      error-handler.ts      Centralized error handling with credential redaction
      request-logger.ts     Structured request/response logging (pino)
    routes/                 Express route handlers
      send.ts               POST /send - send iMessage
      conversations.ts      GET /conversations, GET /conversations/:id
      health.ts             GET /health - service health check
    services/               Business logic
      bb-client.ts          BlueBubbles REST API client (singleton)
      rate-limiter.ts       Token bucket rate limiter with jitter
      health-check.ts       Deep health check (BB status, iPhone, iMessage)
      health-monitor.ts     Periodic polling and downtime alerting
      webhook-relay.ts      Inbound message relay to CRM webhook
      retry-queue.ts        Exponential backoff retry for failed webhooks
      bb-events.ts          Socket.IO event listener for BB webhooks
      backfill.ts           Backfill missed messages on reconnect
      dedup-buffer.ts       Deduplication buffer for BB events
      sync-state.ts         last_synced_at persistence (local JSON)
    types/                  TypeScript types and error codes
      errors.ts             AppError class and error code enum
      bb-types.ts           BlueBubbles API response types
      api-types.ts          API request/response DTOs
    app.ts                  Express app factory
    server.ts               Server entry point (starts Express, PM2, services)
  deploy/                   Deployment configuration
    nginx/
      bluebubbles-api.conf  Nginx reverse proxy template
    pm2-startup.sh          PM2 launchd startup script
  docs/                     Documentation
    API.md                  Full API reference
    DEPLOYMENT.md           AWS EC2 Mac deployment guide
    ONBOARDING.md           Adding new phone numbers
    HANDOFF.md              This file
  dist/                     Compiled JavaScript output (generated by npm run build)
  ecosystem.config.js       PM2 multi-instance configuration
  .env.example              Environment variable template with all vars documented
```

## Build, Test, Run

| Command | Purpose |
|---------|---------|
| `npm install` | Install dependencies |

## Key Architecture Decisions

### Fire-and-forget send

POST /send returns immediately with a `messageId` (crypto.randomUUID) and `status: "queued"`. The actual BlueBubbles send happens asynchronously with jitter delay. This prevents Apple from flagging rapid sequential messages as spam and keeps API response times fast.

### Singleton services

BlueBubbles client, rate limiter, and dedup buffer use a factory/singleton pattern (`getRateLimiter()`, `getBBClient()`, `getDedup()`). This ensures one shared instance across the application, initialized once at startup.

### Webhook relay

Inbound messages flow through: Socket.IO listener -> dedup buffer -> relay to CRM webhook URL. The dedup buffer prevents duplicate deliveries (BlueBubbles sends 2-3 events per message). Failed deliveries enter the retry queue.

### Retry queue with exponential backoff

Failed webhook deliveries are queued and retried with exponential backoff. The queue is bounded (`RETRY_QUEUE_MAX_SIZE`) to prevent unbounded memory growth. Processing uses a setTimeout chain (not setInterval) to avoid overlap.

### Backfill on reconnect

When the service starts or reconnects to BlueBubbles, it queries for messages since `last_synced_at` and fires them to the CRM webhook. This ensures no messages are lost during downtime. The `last_synced_at` timestamp is stored in a local JSON file using atomic temp-file-then-rename writes.

### Health monitoring

Periodic polling with an alert state machine: consecutive failures are counted, and a single alert fires to the alert webhook URL after the configured threshold (`ALERT_AFTER_FAILURES`). The alert resets when BlueBubbles recovers.

### Express binds 127.0.0.1 only

The Express server listens only on localhost. All public traffic goes through the nginx reverse proxy, which handles HTTPS/SSL termination, rate limiting (10r/s burst=20), and security headers.

### No database

The only persisted state is `last_synced_at` in a local JSON file. Tyler's CRM handles all message persistence. This keeps the service stateless and simple.
