**1. Title and overview**: "BlueBubbles iMessage API Reference". Base URL: `https://YOUR_DOMAIN` (replace with actual domain). All requests/responses are JSON.

**2. Authentication section (per D-03)**:
- All endpoints except GET /health require `Authorization: Bearer YOUR_API_KEY` header.
- Missing or invalid key returns 401 with `AUTH_FAILURE` error code.
- API_KEY must be at least 16 characters.

**3. Endpoints section (per D-01)** — document each endpoint with:
- Method and path
- Description
- Request format (body for POST, query params for GET)
- Response format with exact JSON shape
- Error responses that can occur
- curl example with -H "Authorization: Bearer ..." header

**POST /send**:
- Request body: `{ "to": "+15551234567", "message": "Hello from API" }`
- `to`: phone number (any format, auto-normalized to E.164)
- `message`: 1-5000 characters
- Success 200: `{ "messageId": "uuid", "status": "queued" }`
- Note: "queued" means accepted, not delivered. Delivery confirmed via webhook.
- Errors: VALIDATION_ERROR (400), INVALID_PHONE (400), RATE_LIMITED (429), BB_OFFLINE (503), INTERNAL_ERROR (500)

**GET /conversations**:
- Query params: `offset` (default 0), `limit` (default 25, max 100)
- Response: `{ "data": [{ "id", "contact", "lastMessage", "timestamp", "unreadCount" }], "pagination": { "offset", "limit", "total" } }`
- Errors: VALIDATION_ERROR (400)

**GET /conversations/:id**:
- Path param: `:id` is the chat GUID from BlueBubbles
- Query params: same pagination as above
- Response: `{ "data": [{ "id", "sender", "body", "timestamp", "isFromMe" }], "pagination": { "offset", "limit", "total" } }`
- Errors: VALIDATION_ERROR (400)

**GET /health** (public, no auth required):
- No parameters
- Response: health check result with BlueBubbles status, iPhone connection, iMessage auth, macOS version, BB version, lastChecked timestamp
- Always returns 200

**4. Error Code Reference Table (per D-02)**:
Table with columns: Code | HTTP Status | Retryable | Description
Document all 9 error codes from ERROR_CODES:
- VALIDATION_ERROR | 400 | No | Request body or query params failed validation
- INVALID_PHONE | 400 | No | Phone number could not be normalized to E.164
- RATE_LIMITED | 429 | Yes | Too many requests, retry after delay
- BB_OFFLINE | 503 | Yes | BlueBubbles server is not reachable
- BB_IMESSAGE_DISCONNECTED | 503 | Yes | iMessage is signed out or iPhone disconnected
- SEND_FAILED | 500 | Yes | BlueBubbles accepted request but send failed
- INTERNAL_ERROR | 500 | No | Unexpected server error
- WEBHOOK_DELIVERY_FAILED | N/A | Yes | CRM webhook delivery failed (internal, not returned to API clients)
- AUTH_FAILURE | 401 | No | Missing or invalid API key

**5. Error response format section**:
All errors return: `{ "error": { "message": "...", "code": "ERROR_CODE", "retryable": true|false } }`
Note: 500+ errors return generic "Internal server error" message (credential protection).

**6. Webhook Events section**:
Document what Tyler's CRM receives:
- Inbound message webhook: fires to CRM_WEBHOOK_URL with sender, body, timestamp, threadId
- Delivery confirmation: forwarded when BB reports message delivered
- Deduplication: duplicate BB events produce only one webhook fire
- Retry: failed deliveries retried with exponential backoff

**7. Rate Limiting section**:
- App-level token bucket: configurable capacity and refill rate
- Nginx-level: 10 req/s with burst of 20
- Rate limited requests get 429 with RATE_LIMITED code
